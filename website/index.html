The real problem

You can build faster than you can understand.

This isn’t a documentation problem. The app runs. The feature looks finished. The gap is understanding — and it widens every session.

  1. 01

    The AI ships it

    You describe what you want, accept the diffs, and keep going. Pretty soon the software exists and it mostly works.

  2. 02

    The codebase outgrows your map of it

    Prompts, templates, and agent sessions pile up. The repository grows faster than your mental model of it.

  3. 03

    Every change becomes a guess

    You don’t know which files matter, what’s verified, or what the next edit breaks. And the next AI session knows even less than you do.

“I know the app works, but I don’t understand how. The AI changed six files and I can’t tell which ones matter.”

What it creates

One folder your code carries: .vibekb/

Not a database. Not a cloud service. A structured, human-readable model committed alongside your code — the single source of truth for what the software is doing right now.

.vibekb/

  • manifest.jsonprovenance — the commit analysed
  • project/identity, intent, current state, constraints
  • functionality/the primary unit — what the software does
  • system/components, flows, storage, deployment
  • files/the files that actually matter
  • diagrams/explainable, source-grounded topology
  • memory/decisions, warnings, discoveries, changes
  • work/current AI work & the next-agent handoff

Each part answers a question you actually ask:

  • What does it do? Functionality records, each with an honest status — implemented, partial, broken, or unknown.
  • How does it work? Systems, flows, and the files behind each behaviour.
  • What’s risky? Warnings and constraints, linked to the behaviour they threaten.
  • What is AI changing? The current objective, its blast radius, and how it’ll be verified.
  • Where do I start? A handoff written for the next human or agent.

And it renders — as a live guide you can browse, or a static site you can publish. Open VibeKB’s own guide →

How it works

Five honest steps. No magic background analysis.

VibeKB detects that code changed. It never claims to understand a change on its own — your coding agent does the reading, following a repository-owned workflow.

  1. 1

    Install the CLI

    One command puts vibekb on your PATH. No PHP, no database, nothing touched in your app.

  2. 2

    Run it in your repository

    vibekb install . scaffolds an empty-but-valid .vibekb/ workspace. It writes only under that folder.

  3. 3

    Let your agent build the model

    Point Claude Code, Cursor, Codex, or another agent at the integration prompt. It reads your source and writes the first model — distinguishing implemented from inferred.

  4. 4

    Keep it in step as the code changes

    vibekb check flags drift — broken references, changed files, stale snapshots — so the model never quietly rots.

  5. 5

    Read it before you edit

    Browse the guide, or hand it to the next AI session as durable context. Change less by guesswork.

Safe by design

It’s safe to install in a repository you care about.

VibeKB owns exactly one thing — the .vibekb/ folder. Everything else is yours, and stays yours.

What it does

  • Writes everything under .vibekb/
  • Touches shared files (AGENTS.md, CLAUDE.md) only through a single, clearly-marked managed block
  • Lets you preview every change: vibekb install --dry-run .
  • Uninstalls cleanly and ownership-aware: vibekb uninstall .

What it never does

  • Modify your application code during install
  • Replace your README, AGENTS.md, or CLAUDE.md
  • Write anything at your repository root by default
  • Run PHP, call an API, or phone home to install

Install

Two commands to a modelled repository.

1 · Install the CLI (macOS / Linux)

$ curl -fsSL https://iainreid.dev/vibekb/install.sh | sh

2 · Install it into your project

$ vibekb install .

Then open the project in your coding agent and ask it to build the first model using prompts/INTEGRATE_VIBEKB.md.

Other ways to get it

  • Windows — download vibekb-windows-amd64.exe (or -arm64) from GitHub Releases, rename to vibekb.exe, add to PATH.
  • Any platform — grab a binary from GitHub Releases.
  • From source — go build -o vibekb ./cmd/vibekb (Go 1.24+), then ./vibekb install .

Requirements: PHP 8.2+ to run the installed guide & model commands. No database, no AI API, no Docker, no build step. Homebrew & Winget are on the roadmap.
